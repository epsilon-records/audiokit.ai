# AudioKit Development Rules

## Core Development Rules

1. Always use best practices
   - Follow PEP 8 style guide
   - Use type hints consistently
   - Write comprehensive docstrings
   - Keep functions focused and small
   - Use meaningful variable names
   - Handle errors gracefully
   - Write tests for all functionality
   - Use async/await for I/O operations
   - Follow SOLID principles
   - Use dependency injection where appropriate
   - Use ruff for all formatting and linting (replaces flake8, isort, and black)

2. Always use Poetry for dependency management
   - Never use setup.py (deprecated)
   - Maintain pyproject.toml for all configuration
   - Use poetry.lock for deterministic builds
   - Specify dependency ranges appropriately
   - Group dev dependencies separately
   - Use virtual environments consistently
   - Run poetry update regularly
   - Document dependency changes

3. Always use latest stable versions
   - Python 3.11+ for development
   - Latest stable versions of dependencies
   - Regular dependency updates
   - Security patches prioritized
   - Maintain compatibility matrix
   - Document version requirements
   - Use CI to test multiple versions

## Environment Setup

### Development
```bash
# Create new project
poetry new project-name

# Install dependencies
poetry install

# Add dependency
poetry add package-name

# Add dev dependency
poetry add --group dev package-name

# Update dependencies
poetry update

# Run tests
poetry run pytest

# Format and lint code
poetry run ruff check .
poetry run ruff format .
```

### Production
```bash
# Install production dependencies
poetry install --no-dev

# Build package
poetry build

# Publish package
poetry publish
```

AudioKit Development Standards
==============================

Code Standards
--------------
1. Python 3.10+ type hints required
2. FastAPI endpoints must include OpenAPI docs
3. Pydantic models for all I/O
4. 100% test coverage for security-critical code

Workflow
--------
1. Feature branches prefixed with:
   - feat/ - New functionality
   - fix/ - Bug repairs
   - refactor/ - Code improvements
2. PRs require:
   - Architecture review
   - Security assessment
   - Performance impact analysis

Testing
-------
1. Unit tests for all business logic
2. Integration tests for API endpoints
3. Benchmark tests for audio processing

Documentation
-------------
1. Keep OpenAPI spec updated
2. Maintain developer quickstart guide
3. Publish API schema with each release
